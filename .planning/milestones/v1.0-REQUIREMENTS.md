# Requirements: Token Bucket Rate Limiter

**Defined:** 2026-03-31
**Core Value:** Protect API endpoints from abuse while maintaining excellent developer experience through clear error messages, proper HTTP headers, and predictable behavior.

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Core Rate Limiting

- [x] **CORE-01**: Token bucket algorithm with continuous refill (not fixed window)
- [x] **CORE-02**: Per-IP client identification with X-Forwarded-For header parsing
- [x] **CORE-03**: Per-endpoint rate limit configuration (different limits per route)
- [x] **CORE-04**: In-memory storage for rate limit state
- [x] **CORE-05**: HTTP 429 Too Many Requests responses when limit exceeded

### HTTP Headers & Responses

- [x] **HTTP-01**: X-RateLimit-Limit header in all responses
- [x] **HTTP-02**: X-RateLimit-Remaining header in all responses
- [x] **HTTP-03**: X-RateLimit-Reset header in all responses (seconds until reset)
- [x] **HTTP-04**: Retry-After header in 429 responses (seconds to wait)
- [x] **HTTP-05**: Detailed JSON error message in 429 response body

### Configuration & Integration

- [x] **CONF-01**: FastAPI middleware integration (automatic application to all routes)
- [x] **CONF-02**: Environment variable configuration for rate limits
- [x] **CONF-03**: Per-endpoint limit overrides via configuration
- [x] **CONF-04**: Exemption list for operational endpoints (/health, /docs, /redoc, /openapi.json)

### Storage & State Management

- [x] **STOR-01**: In-memory storage implementation using Python dict
- [x] **STOR-02**: Thread-safe atomic operations (asyncio.Lock for async FastAPI)
- [x] **STOR-03**: Memory cleanup with TTL for expired buckets (background task)

### File Structure & Code Quality

- [x] **STRUC-01**: Dedicated middleware directory: app/middleware/rate_limit/
- [x] **STRUC-02**: Modular components: TokenBucket, InMemoryStorage, RateLimiter, Middleware
- [x] **STRUC-03**: Full type hints with Pydantic models for configuration
- [x] **STRUC-04**: Clean separation: bucket logic, storage, service layer, middleware layer
- [x] **STRUC-05**: Integration with existing domain exception patterns (BaseAppException)

### Testing & Quality Assurance

- [x] **TEST-01**: Happy path tests (requests within limit pass through)
- [x] **TEST-02**: Limit exceeded tests (429 responses with correct headers)
- [x] **TEST-03**: Bucket refill tests (time-based token replenishment)
- [x] **TEST-04**: Concurrent request tests (race condition prevention with asyncio.gather)
- [x] **TEST-05**: X-Forwarded-For spoofing tests (security validation)
- [x] **TEST-06**: Load/performance testing (10K+ requests per second)
- [x] **TEST-07**: Memory leak testing (1+ hour run with rotating IPs)

### Documentation

- [x] **DOCS-01**: Update AGENTS.md with rate limiting architecture and patterns
- [x] **DOCS-02**: Configuration examples in documentation (env vars, per-endpoint overrides)
- [x] **DOCS-03**: Exemption endpoint configuration guide

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Per-API-key rate limiting | No authentication system exists; IP-based is sufficient for v1 |
| Rate limit dashboard UI | Environment configuration is sufficient; operators don't need GUI |
| Dynamic rate adjustment | Static limits are simpler; no evidence of need for adaptive limits |
| Queue-based rate limiting | Token bucket provides same protection with simpler implementation |
| Fixed-window algorithm | Allows boundary burst attacks (2× rate bypass); token bucket prevents this |
